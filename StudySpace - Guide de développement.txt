## 1. Workflow de développement

### 1.1 Structure du projet

```
studyspace/
├── apps/
│   ├── web/                    # Next.js frontend
│   │   ├── app/                # App router
│   │   ├── components/         # Composants React
│   │   │   ├── ui/             # shadcn/ui components
│   │   │   ├── workspace/      # Composants workspace
│   │   │   ├── auth/           # Composants auth
│   │   │   └── shared/         # Composants partagés
│   │   ├── lib/                # Utilitaires
│   │   ├── hooks/              # Custom hooks
│   │   ├── styles/             # Styles globaux
│   │   └── public/             # Assets statiques
│   └── api/                    # Backend Node.js
│       ├── src/
│       │   ├── routes/         # Routes API
│       │   ├── controllers/    # Controllers
│       │   ├── services/       # Business logic
│       │   ├── models/         # Modèles de données
│       │   ├── middleware/     # Middlewares
│       │   ├── utils/          # Utilitaires
│       │   └── sockets/        # WebSocket handlers
│       └── tests/              # Tests
├── packages/
│   ├── types/                  # Types TypeScript partagés
│   ├── config/                 # Configs partagées
│   └── ui/                     # Design system exportable
├── docs/                       # Documentation
└── README.md
```

**Règles :**
- Pas de logique métier dans `/components`
- Les services ne dépendent pas des controllers
- Types partagés dans `/packages/types`
- Un dossier = une responsabilité claire

### 1.2 Branching strategy

**Branches permanentes :**
- `main` : production, toujours stable, déployée automatiquement
- `develop` : intégration continue, pre-production

**Branches temporaires :**
- `feature/<ticket-id>-<description>` : nouvelles fonctionnalités
- `bugfix/<ticket-id>-<description>` : corrections de bugs
- `hotfix/<ticket-id>-<description>` : correctifs urgents en production
- `release/<version>` : préparation de release

**Workflow :**
```
feature branch → develop → release branch → main
                    ↓
              tests automatiques
                    ↓
              code review obligatoire
```

**Règles :**
- Jamais de commit direct sur `main` ou `develop`
- Une branche = un ticket/une feature
- Durée de vie max d'une branche : 5 jours
- Merge via Pull Request uniquement
- Squash commits avant merge sur `develop`

### 1.3 Processus de création de features

**Étape 1 : Création de la branche**
```bash
git checkout develop
git pull origin develop
git checkout -b feature/SS-123-video-toggle
```

**Étape 2 : Développement**
- Commits atomiques et descriptifs
- Lint et format avant chaque commit
- Tests unitaires écrits en parallèle
- Push régulier (au moins 1x/jour)

**Étape 3 : Pull Request**

Template PR :
```markdown
## Description
[SS-123] Implémentation du toggle vidéo dans la barre de visio

## Type de changement
- [x] Feature
- [ ] Bugfix
- [ ] Refactoring

## Checklist
- [x] Tests unitaires ajoutés
- [x] Tests e2e mis à jour
- [x] Documentation mise à jour
- [x] Pas de warning lint
- [x] Build successful
- [x] Testé manuellement

## Screenshots
[Si UI] Ajouter avant/après
```

**Étape 4 : Review**
- Au moins 1 approbation requise
- CI doit être verte
- Pas de conflits de merge

**Étape 5 : Merge**
- Squash and merge sur `develop`
- Supprimer la branche après merge

### 1.4 Code review

**Critères obligatoires :**

✅ **Fonctionnel**
- Le code fait ce qu'il doit faire
- Pas de régression
- Edge cases gérés

✅ **Qualité**
- Lisible et maintenable
- Pas de code dupliqué
- Respect des conventions

✅ **Performance**
- Pas de re-renders inutiles (React)
- Requêtes optimisées
- Bundle size raisonnable

✅ **Sécurité**
- Pas d'injection SQL/XSS
- Validation des inputs
- Auth/permissions vérifiées

**Process de review :**

1. **Reviewer :** Lire la description du PR
2. **Reviewer :** Checker le code ligne par ligne
3. **Reviewer :** Tester localement si UI
4. **Reviewer :** Laisser des commentaires constructifs
5. **Auteur :** Répondre et corriger
6. **Reviewer :** Approuver ou demander des changements

**Exemples de commentaires :**

❌ **Mauvais :**
```
"Ce code est nul"
"Pourquoi tu fais ça ?"
```

✅ **Bon :**
```
"Cette fonction pourrait être simplifiée en utilisant Array.reduce(). 
Qu'en penses-tu ?"

"Ce composant fait 200 lignes. On pourrait extraire la logique de 
formulaire dans un hook custom ?"

"N'oublie pas de gérer le cas où l'utilisateur n'a pas de permission. 
Exemple : if (!hasPermission) return <AccessDenied />"
```

**Règles :**
- Reviewer sous 24h maximum
- Être constructif, jamais agressif
- Proposer des solutions, pas juste critiquer
- Approuver si les changements demandés sont mineurs

### 1.5 QA et tests

**Hiérarchie des tests :**

1. **Tests unitaires** (70%)
   - Fonctions pures
   - Hooks custom
   - Utilitaires
   - Services métier

2. **Tests d'intégration** (20%)
   - API endpoints
   - Synchronisation WebSocket
   - Flux de données

3. **Tests e2e** (10%)
   - Parcours utilisateur critiques
   - Création de salle
   - Session de travail complète

**Outils :**
- **Unitaires :** Vitest
- **Composants :** React Testing Library
- **E2E :** Playwright

**Structure des tests :**
```typescript
describe('WorkspaceService', () => {
  describe('createWorkspace', () => {
    it('should create workspace with valid data', async () => {
      // Arrange
      const data = { name: 'Test', tag: 'maths' };
      
      // Act
      const result = await service.createWorkspace(data);
      
      // Assert
      expect(result.id).toBeDefined();
      expect(result.name).toBe('Test');
    });

    it('should throw error with invalid tag', async () => {
      const data = { name: 'Test', tag: 'invalid' };
      
      await expect(service.createWorkspace(data))
        .rejects.toThrow('Invalid tag');
    });
  });
});
```

**Critères d'acceptance :**
- Couverture > 80% sur le code critique
- Tous les tests passent avant merge
- Pas de tests flaky
- Tests e2e critiques verts

**Règles :**
- Un test = un comportement
- Tests indépendants (pas d'ordre requis)
- Noms descriptifs
- Pas de logique dans les tests

### 1.6 Déploiement et versioning

**Versioning sémantique (SemVer) :**
```
MAJOR.MINOR.PATCH
  1  .  2  .  3

MAJOR: Breaking changes
MINOR: Nouvelles features (backward compatible)
PATCH: Bug fixes
```

**Exemples :**
- `1.0.0` : Premier release MVP
- `1.1.0` : Ajout de l'éditeur d'équations
- `1.1.1` : Fix bug synchronisation
- `2.0.0` : Refonte complète de l'UI

**Pipeline CI/CD (GitHub Actions) :**

```
develop → Lint → Tests → Build → Deploy Staging
main → Lint → Tests → Build → Deploy Production
```

**Étapes du pipeline :**
1. Lint & format check
2. Tests unitaires
3. Tests e2e (staging uniquement)
4. Build
5. Deploy (Vercel frontend + Railway backend)
6. Health check
7. Notification (Discord/Slack)

**Release cycle :**
- **Staging :** Deploy automatique à chaque merge sur `develop`
- **Production :** Release manuelle tous les vendredis 18h
- **Hotfix :** Deploy immédiat si critique

**Rollback :**
- Possible en 1 clic via Vercel/Railway dashboard
- Garder les 5 derniers builds disponibles
- Documentation du rollback dans un incident report

## 2. Conventions de nommage

### 2.1 Variables et constantes

**Variables locales :** `camelCase`
```typescript
const userId = 123;
const workspaceName = 'Maths L3';
const isActive = true;
```

**Constantes globales :** `UPPER_SNAKE_CASE`
```typescript
const MAX_FILE_SIZE = 50_000_000; // 50MB
const DEFAULT_WORKSPACE_TAG = 'general';
const API_BASE_URL = process.env.NEXT_PUBLIC_API_URL;
```

**Booléens :** Préfixer avec `is`, `has`, `can`, `should`
```typescript
const isLoading = true;
const hasPermission = false;
const canEdit = true;
const shouldSync = false;
```

**Collections :** Pluriel
```typescript
const users = [];        // ✅
const user = [];         // ❌

const workspaceList = [];  // ❌ (redondant)
const workspaces = [];     // ✅
```

### 2.2 Fonctions

**Functions :** `camelCase`, verbe à l'infinitif
```typescript
function createWorkspace() {}
function deleteUser() {}
function validateEmail() {}
function handleSubmit() {}
```

**Event handlers :** Préfixer avec `handle` ou `on`
```typescript
function handleClick() {}
function handleInputChange() {}
function onWorkspaceCreated() {}
```

**Getters :** Préfixer avec `get`
```typescript
function getUserById(id: string) {}
function getActiveWorkspaces() {}
```

**Predicates (retournent boolean) :** `is`, `has`, `can`
```typescript
function isValidEmail(email: string): boolean {}
function hasAccess(userId: string): boolean {}
function canEditWorkspace(userId: string): boolean {}
```

**Async functions :** Pas de préfixe spécial
```typescript
async function fetchWorkspaces() {}           // ✅
async function asyncFetchWorkspaces() {}      // ❌ (redondant)
```

### 2.3 Classes et types

**Classes :** `PascalCase`
```typescript
class WorkspaceService {}
class UserController {}
class SocketManager {}
```

**Interfaces :** `PascalCase`, pas de préfixe `I`
```typescript
interface User {}                // ✅
interface IUser {}               // ❌

interface WorkspaceConfig {}     // ✅
```

**Types :** `PascalCase`
```typescript
type UserId = string;
type WorkspaceTag = 'maths' | 'info' | 'physique';
type ApiResponse<T> = { data: T; error?: string };
```

**Enums :** `PascalCase` pour le nom, `UPPER_SNAKE_CASE` pour les valeurs
```typescript
enum WorkspaceRole {
  OWNER = 'OWNER',
  MEMBER = 'MEMBER',
  GUEST = 'GUEST'
}

enum HttpStatus {
  OK = 200,
  BAD_REQUEST = 400,
  UNAUTHORIZED = 401
}
```

### 2.4 Fichiers et dossiers

**Composants React :** `PascalCase.tsx`
```
WorkspaceCard.tsx
VideoToggle.tsx
EquationEditor.tsx
```

**Hooks :** `camelCase.ts`, préfixer avec `use`
```
useWorkspace.ts
useAuth.ts
useRealtime.ts
```

**Utilitaires :** `camelCase.ts`
```
formatDate.ts
validateEmail.ts
storage.ts
```

**Services :** `camelCase.service.ts`
```
workspace.service.ts
auth.service.ts
socket.service.ts
```

**Types :** `camelCase.types.ts`
```
workspace.types.ts
user.types.ts
api.types.ts
```

**Tests :** Même nom + `.test.ts` ou `.spec.ts`
```
WorkspaceCard.test.tsx
workspace.service.test.ts
formatDate.spec.ts
```

**Dossiers :** `kebab-case`
```
/workspace-manager
/video-controls
/equation-editor
```

### 2.5 Composants React

**Composants :** `PascalCase`
```tsx
export function WorkspaceCard() {}
export function VideoToggle() {}
export default function EquationEditor() {}
```

**Props interface :** Nom du composant + `Props`
```typescript
interface WorkspaceCardProps {
  workspace: Workspace;
  onDelete: () => void;
}

function WorkspaceCard({ workspace, onDelete }: WorkspaceCardProps) {}
```

**Composants composés :** Point notation
```tsx
const Workspace = {
  Card: WorkspaceCard,
  List: WorkspaceList,
  Header: WorkspaceHeader,
};

// Usage
<Workspace.Card />
<Workspace.List />
```

### 2.6 Endpoints API

**REST :**
```
GET    /api/workspaces
GET    /api/workspaces/:id
POST   /api/workspaces
PUT    /api/workspaces/:id
DELETE /api/workspaces/:id

GET    /api/workspaces/:id/sessions
POST   /api/workspaces/:id/sessions
GET    /api/workspaces/:id/members
POST   /api/workspaces/:id/members
DELETE /api/workspaces/:id/members/:userId
```

**Règles :**
- Pluriel pour les collections
- `kebab-case` pour les segments composés
- Pas de verbe dans l'URL (utiliser HTTP verbs)
- Versioning dans l'URL : `/api/v1/...`

**Query params :** `camelCase`
```
GET /api/workspaces?tagFilter=maths&sortBy=createdAt&limit=20
```

**WebSocket events :** `kebab-case` avec préfixe namespace
```typescript
// Émission
socket.emit('workspace:updated', data);
socket.emit('member:joined', data);
socket.emit('canvas:draw', data);

// Réception
socket.on('workspace:updated', handler);
socket.on('member:joined', handler);
```

### 3.2 Quand commenter

**✅ COMMENTER :**

**1. Logique complexe**
```typescript
// Utilise l'algorithme de Bresenham pour tracer une ligne optimisée
function drawLine(x0, y0, x1, y1) {
  const dx = Math.abs(x1 - x0);
  const dy = Math.abs(y1 - y0);
  // ...
}
```

**2. Raisons non-évidentes**
```typescript
// Timeout nécessaire pour éviter un race condition avec Daily.co SDK
setTimeout(() => initializeVideo(), 100);
```

**3. Limitations connues**
```typescript
// LIMITATION: Ne supporte que les PDF < 50MB pour des raisons de performance
function uploadFile(file: File) {
  if (file.size > MAX_FILE_SIZE) {
    throw new Error('File too large');
  }
}
```

**4. Workarounds temporaires**
```typescript
// WORKAROUND: Daily.co ne supporte pas le mute programmatique en v1.2
// À retirer quand la v1.3 sera stable
function muteAllParticipants() {
  participants.forEach(p => p.setAudio(false));
}
```

**❌ NE PAS COMMENTER :**

**1. Code évident**
```typescript
// Incrémente le compteur ❌
counter++;

// Retourne la liste des utilisateurs ❌
function getUsers() {
  return users;
}
```

**2. Redire le code**
```typescript
// Set isLoading to true ❌
setIsLoading(true);
```

**3. Commentaires obsolètes**
```typescript
// Cette fonction est dépréciée ❌
// Utilise fetchWorkspacesV2() à la place ❌
// (Si c'est déprécié, supprime-le ou utilise @deprecated)
```

## 6. Principes UX

### 6.1 Patterns autorisés/interdits

**✅ AUTORISÉS :**

**Confirmation avant action destructive**
```
Supprimer un workspace → Modal de confirmation
"Êtes-vous sûr de vouloir supprimer [Nom] ?"
[Annuler] [Supprimer]
```

**Feedback immédiat**
```
Upload fichier → Progress bar
Sauvegarde → Toast "Sauvegardé" (2 sec)
Erreur → Toast error persistant jusqu'à correction
```

**Empty states informatifs**
```
Pas de workspace → 
"Créez votre premier espace de travail"
[Bouton CTA]
```

**Progressive disclosure**
```
Options avancées masquées par défaut
Accessible via "Options avancées >"
```

**❌ INTERDITS :**

**Supprimer sans confirmation**
```
❌ Clic sur poubelle → suppression immédiate
✅ Clic sur poubelle → modal de confirmation
```

**Loader sans timeout**
```
❌ Spinner infini si l'API ne répond pas
✅ Timeout après 30 sec + message d'erreur
```

**Modals en cascade**
```
❌ Modal qui ouvre une autre modal
✅ Fermer la première avant d'ouvrir la seconde
```

**Auto-scroll agressif**
```
❌ Scroll automatique qui interrompt la lecture
✅ Scroll doux et uniquement si nécessaire
```

**Popups non-sollicités**
```
❌ Modal "Notez-nous !" après 30 secondes
✅ Demander après une action réussie (création workspace)
```

### 6.2 Navigation

**Profondeur maximum : 3 niveaux**
```
Home → Workspace → Session
```

**Règles de cohérence :**

1. **Breadcrumb obligatoire si profondeur > 2**
```
Accueil > Mes workspaces > Maths L3
```

2. **Bouton retour toujours visible**
```
← Retour (en haut à gauche)
```

3. **Navigation principale toujours accessible**
```
Sidebar ou top bar fixe
```

4. **URLs lisibles et partageable**
```
✅ /workspace/abc123/session/def456
❌ /app?id=abc123&view=session&sid=def456
```

**Navigation workspace :**
```
Sidebar (desktop) :
- Liste des workspaces
- Bouton "Créer"
- Historique récent

Bottom bar (mobile) :
- Home
- Mes workspaces
- Créer
- Profil
```

**États de navigation :**
- Actif : bg-primary-50, text-primary-600
- Hover : bg-neutral-100
- Disabled : opacity-50, cursor-not-allowed

### 6.3 Feedback utilisateur

**Types de feedback :**

**1. Immédiat (< 100ms)**
```
Clic sur bouton → Changement visuel (active state)
Hover → Changement de curseur
Input focus → Border highlight
```

**2. Court terme (< 1 sec)**
```
Sauvegarde réussie → Toast "Sauvegardé ✓"
Copie lien → Toast "Lien copié"
Ajout membre → Toast "Membre ajouté"
```

**3. Long terme (> 1 sec)**
```
Upload fichier → Progress bar + %
Génération PDF → "Génération en cours..." + spinner
```

**Timing des toasts :**
```
Success: 2 secondes
Info:    3 secondes
Warning: 5 secondes
Error:   Persistant (fermeture manuelle)
```

**Wording :**

**✅ Bon :**
```
"Workspace créé avec succès"
"Impossible de supprimer ce fichier"
"Connexion perdue. Reconnexion automatique..."
```

**❌ Mauvais :**
```
"Success!" (pas assez spécifique)
"Error 500" (technique, pas user-friendly)
"Oops!" (pas professionnel)
```

**Position des toasts :**
```
Desktop: top-right
Mobile:  bottom (au-dessus du bottom bar)
```

### 6.4 Animations et transitions

**Durées :**
```
Ultra rapide: 100ms  (Hover states)
Rapide:       200ms  (Dropdowns, tooltips)
Standard:     300ms  (Modals, slides)
Lent:         500ms  (Page transitions)
```

**Easing :**
```
ease-in:     Accélération (fermetures)
ease-out:    Décélération (ouvertures) ← Par défaut
ease-in-out: Doux (mouvements larges)
```

**Règles :**

**✅ Animer :**
- Ouverture/fermeture de modals
- Apparition de dropdowns
- Transitions de page
- Progress indicators
- Hover states subtils

**❌ Ne pas animer :**
- Texte (sauf scroll)
- Formulaires (distraction)
- Contenu critique (lecture)
- Elements > 50% de l'écran

**Préférence utilisateur :**
```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

**Exemples d'animations :**

**Modal :**
```
Entrée: opacity 0→1 + scale 0.95→1 (300ms ease-out)
Sortie: opacity 1→0 + scale 1→0.95 (200ms ease-in)
```

**Toast :**
```
Entrée: slide-in-right + fade-in (200ms)
Sortie: fade-out (150ms)
```

**Dropdown :**
```
Entrée: opacity 0→1 + translateY(-10px→0) (200ms)
```

### 6.5 Formulaires

**Structure :**

**1. Label toujours visible**
```
Pas de placeholder comme label
Label au-dessus de l'input
Font-medium, text-sm
```

**2. Helper text si besoin**
```
Sous l'input, text-xs, text-neutral-600
Ex: "Minimum 8 caractères"
```

**3. Erreur inline**
```
Sous l'input, text-xs, text-error-500
Icône d'erreur dans l'input
Border rouge
```

**4. Validation en temps réel (mais pas trop tôt)**
```
✅ Valider après que l'user quitte le champ (onBlur)
❌ Valider à chaque keystroke (trop agressif)
```

**Exemple de formulaire :**
```
[Label]
Nom du workspace *

[Input]
┌─────────────────────────────┐
│ Maths L3                    │
└─────────────────────────────┘

[Helper]
Choisissez un nom descriptif

[Erreur si invalide]
⚠ Le nom doit faire au moins 3 caractères
```

**Règles :**

- Champs obligatoires : astérisque (*) après le label
- Grouper les champs liés visuellement
- Maximum 5-7 champs par page
- Bouton submit toujours visible (sticky si long formulaire)
- Auto-focus sur le premier champ
- Désactiver le submit si formulaire invalide
- Feedback de succès après soumission

**États des inputs :**
```
Default:  border-neutral-300
Focus:    border-primary-500 + ring
Error:    border-error-500 + ring-error
Success:  border-success-500 (optionnel)
Disabled: bg-neutral-100 + cursor-not-allowed
```

### 6.6 Gestion des erreurs

**Hiérarchie des erreurs :**

**1. Erreurs de validation (user fixable)**
```
Message: Clair et actionnable
Position: Inline sous le champ
Exemple: "L'email doit contenir un @"
```

**2. Erreurs système (user non-responsable)**
```
Message: Explique ce qui s'est passé + next step
Position: Toast persistant
Exemple: "Impossible de sauvegarder. Vérifiez votre connexion."
```

**3. Erreurs critiques (blocantes)**
```
Message: Explique + propose solution
Position: Modal ou full-page
Exemple: "Session expirée. Reconnectez-vous."
```

**Wording des erreurs :**

**✅ Bon :**
```
"Impossible de charger les workspaces. Réessayez dans quelques instants."
"Ce nom est déjà utilisé. Choisissez-en un autre."
"Fichier trop volumineux. Maximum 50 MB."
```

**❌ Mauvais :**
```
"Error 500"
"Invalid input"
"Something went wrong"
```

**Structure message d'erreur :**
```
1. Ce qui s'est passé (court)
2. Pourquoi (si pertinent)
3. Que faire (actionnable)

Exemple:
"Impossible d'envoyer l'invitation.
L'utilisateur existe déjà dans ce workspace.
Vérifiez la liste des membres."
```

**Edge cases à gérer :**

**Perte de connexion :**
```
- Afficher un banner persistant "Hors ligne"
- Désactiver les actions qui nécessitent le réseau
- Queue les actions pour envoi quand connexion revient
- Notifier quand reconnecté
```

**Session expirée :**
```
- Rediriger vers login
- Sauvegarder l'état (draft) si possible
- Message: "Votre session a expiré. Reconnectez-vous."
```

**Permissions insuffisantes :**
```
- Message clair: "Vous n'avez pas la permission de faire ça"
- Masquer les boutons inaccessibles (pas juste disable)
```

**Données introuvables (404) :**
```
- Page dédiée avec message
- Bouton retour à l'accueil
- Suggestions (si applicable)
```

### 6.7 Onboarding

**Première visite :**

**Étape 1 : Welcome screen**
```
"Bienvenue sur StudySpace"
Illustration simple
"Travaillez ensemble comme si vous étiez dans la même salle"
[Commencer]
```

**Étape 2 : Créer premier workspace (guidé)**
```
Modal avec instructions claires
"Créez votre premier espace de travail"
Formulaire pré-rempli avec exemple
[Créer] bouton mis en avant
```

**Étape 3 : Tour interactif (optionnel, skippable)**
```
Tooltips contextuels (max 3-4)
1. "Voici votre tableau blanc"
2. "Invitez des membres ici"
3. "Commencez une session ici"
[Suivant] [Passer]
```

**Règles onboarding :**
- Maximum 3 étapes obligatoires
- Toujours skippable après la première fois
- Jamais bloquer l'utilisation
- Aide contextuelle accessible via "?" icon

**Progressive onboarding :**
```
Au lieu d'un tour complet au début:
- Tooltip quand l'user survole un bouton la 1ère fois
- Empty state avec CTA dans chaque section vide
- "Astuce du jour" discrète (dismissable)
```

**Aide contextuelle permanente :**
```
Icon "?" dans le header
→ Ouvre un drawer avec:
  - FAQ rapide
  - Raccourcis clavier
  - Contact support
```
